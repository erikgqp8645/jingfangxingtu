# 数据层执行清单 v1

## 这份文档是干什么的

这份文档不是概念说明。

这是一份给后续开发和数据整理直接使用的执行清单。

它要解决的问题只有一个：

`在现有“条文 -> 关键词 -> 来源片段”关系图模型已经确定的前提下，后面的数据层到底要怎么一步一步整理。`

如果你只想先记住一句话，请记住：

`后面不要再扩更多节点类型，而是把现有三类节点做标准化、可审计、可落库。`

---

## 一、当前项目已经确定的节点

现在这个项目里，关系图的数据核心已经比较明确，只有三类节点：

1. 条文节点 `clause`
2. 关键词节点 `keyword`
3. 来源片段节点 `source`

当前关系链路也已经比较明确：

`条文 -> 关键词 -> 来源片段`

这意味着后续的数据层整理，不应该再回到“复杂知识图谱”的方向。

后面要做的，不是继续增加很多新节点，而是把这三类节点的数据结构、关系结构、维护流程、质量检查流程做扎实。

---

## 二、数据层第二阶段的总目标

第二阶段的数据层整理，一共要完成 6 件事：

1. 固定三类节点的标准字段。
2. 固定节点之间的关系结构。
3. 固定源数据目录的维护规范。
4. 建立可重复运行的数据审计能力。
5. 建立 SQLite（轻量数据库）运行层。
6. 让后续补数据这件事变成“按清单执行”，而不是“想到哪补到哪”。

换句话说，第二阶段的目标不是“补更多数据”本身，而是：

`先把数据整理的规则和底座搭好，再让补数据这件事越来越快。`

---

## 三、三类节点的正式定义

这一部分非常重要。

如果这一部分不先定死，后面数据越多，返工就越多。

### 3.1 条文节点 Clause

条文节点表示一本经典中的一条正文。

建议统一字段如下：

| 字段名 | 中文说明 | 是否必须 | 作用 |
|------|------|------|------|
| `id` | 全局唯一编号 | 是 | 每条条文唯一标识 |
| `book_id` | 经典编号 | 是 | 表示属于哪本书 |
| `chapter_id` | 章节编号 | 否 | 表示属于哪一章 |
| `display_no` | 展示条号 | 否 | 页面上显示的条号 |
| `title` | 条文标题 | 是 | 左侧列表和正文标题使用 |
| `content` | 原文正文 | 是 | 正文原文区显示 |
| `translation` | 白话解释 | 是 | 右下或详情区显示 |
| `source_file` | 原始文件路径 | 否 | 便于追溯 |
| `status` | 数据状态 | 否 | 便于知道是否已检查 |

建议状态值：

- `draft`：草稿，刚录入
- `keyword_checked`：关键词已检查
- `relation_checked`：关系已检查
- `published`：已可稳定展示

### 3.2 关键词节点 Keyword

关键词节点表示一个可以参与关系建立的词。

建议统一字段如下：

| 字段名 | 中文说明 | 是否必须 | 作用 |
|------|------|------|------|
| `id` | 关键词编号 | 是 | 数据库主键或唯一标识 |
| `name` | 原始关键词 | 是 | 页面展示用 |
| `normalized_name` | 标准化关键词 | 是 | 去重、统计、合并使用 |
| `type` | 关键词类型 | 否 | 症状、病机、脉象等分类 |
| `quality_level` | 质量等级 | 否 | A/B/C/D 级管理 |
| `status` | 审核状态 | 否 | 是否已确认 |
| `source` | 关键词来源 | 否 | 人工、导入、推断 |

建议 `source` 取值：

- `manual`：页面右键添加
- `json`：直接改条文 JSON（数据文件格式）
- `imported`：导入脚本生成
- `inferred`：程序推断

### 3.3 来源片段节点 Source Fragment

来源片段节点不是整本书。

它表示右侧命中列表里真正要展示的那一条内容。

建议统一字段如下：

| 字段名 | 中文说明 | 是否必须 | 作用 |
|------|------|------|------|
| `id` | 片段编号 | 是 | 唯一标识 |
| `source_id` | 来源编号 | 是 | 表示属于哪个来源 |
| `source_name` | 来源名称 | 是 | 页面显示来源名 |
| `category` | 分类 | 否 | 如内经、伤寒、温病 |
| `title` | 篇名或段名 | 是 | 页面显示标题 |
| `content` | 完整片段内容 | 是 | 真实正文内容 |
| `content_excerpt` | 摘要 | 否 | 预览用 |
| `source_type` | 来源类型 | 是 | `json` 或 `txt` |
| `source_file` | 原始文件路径 | 否 | 追溯用 |
| `status` | 整理状态 | 否 | 便于后期持续清洗 |

建议状态值：

- `raw`：原始状态
- `split`：已切段
- `checked`：已检查
- `published`：可稳定使用

## 四、两类关系的正式定义

后面不能只整理节点，还要整理边。

建议把关系固定为两层。

### 4.1 条文-关键词关系

关系语义：

`这条条文包含这个关键词`

建议字段：

| 字段名 | 中文说明 |
|------|------|
| `clause_id` | 条文编号 |
| `keyword_id` | 关键词编号 |
| `source` | 关系来源 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

这个关系后面对应数据库表建议叫：

- `clause_keywords`

### 4.2 来源片段-关键词关系

关系语义：

`这个来源片段命中了这个关键词`

建议字段：

| 字段名 | 中文说明 |
|------|------|
| `fragment_id` | 来源片段编号 |
| `keyword_id` | 关键词编号 |
| `match_type` | 命中方式 |
| `created_at` | 创建时间 |

建议 `match_type` 取值：

- `json_exact`
- `txt_search`
- 后续可扩展 `alias_match`

这个关系后面对应数据库表建议叫：

- `relation_entry_keywords`

## 五、源数据层的整理目标

这一层说的是：

`data/` 目录以后应该整理成什么样，才能长期维护。

### 5.1 总原则

必须固定下面三条：

1. `data/` 永远是源数据。
2. SQLite（轻量数据库）永远是运行副本。
3. 页面上的关系结果，最终都能追溯回 `data/` 里的源文件。

### 5.2 条文源数据目标

目标不是“能显示就行”，而是“后面导入数据库时不返工”。

所以条文 JSON（数据文件格式）需要做到：

1. 编号统一
2. 字段统一
3. 路径统一
4. 状态可追踪
5. 关键词可以独立维护

### 5.3 关联解析源数据目标

目标不是“先有一堆全文”，而是“让右侧命中足够可读”。

所以来源数据需要做到：

1. 来源先登记
2. 文件名和登记名一致
3. JSON 优先
4. TXT 回退
5. 命中单位尽量是片段，不是整本内容

## 六、第二阶段的五个实施阶段

下面是建议的执行顺序。

不要跳阶段。

### 阶段 1：冻结标准

这一阶段先不急着补新数据。

先把标准定死。

必须完成的事项：

1. 固定三类节点字段清单
2. 固定两类关系结构
3. 固定条文编号规则
4. 固定关键词命名规则
5. 固定来源登记规则
6. 固定 JSON / TXT 命名规则

这一阶段交付物：

- 节点字段标准
- 关系字段标准
- 命名规范
- 编号规范

### 阶段 2：全量体检

这一阶段不新增复杂功能。

先看清楚当前数据到底有什么问题。

必须完成的事项：

1. 扫描所有没有关键词的条文
2. 扫描所有有关键词但零命中的条文
3. 扫描所有重复关键词
4. 扫描所有疑似泛词
5. 扫描所有来源登记缺失
6. 扫描所有文件存在但未登记的来源
7. 扫描所有重复来源片段
8. 扫描所有只有 TXT、没有 JSON 的来源

这一阶段交付物：

- 数据体检报告
- 优先补充清单
- 风险来源清单

### 阶段 3：清洗核心数据

这一阶段开始真正整理数据。

建议优先处理：

1. 当前首页和常用经典
2. 当前已有关键词但命中质量差的条文
3. 完全没有关键词的核心条文
4. 高频来源

必须完成的事项：

1. 优先补条文关键词
2. 优先把高价值 TXT 整理成 JSON
3. 清洗低质量关键词
4. 控制每条条文关键词密度
5. 统一来源片段长度

这一阶段交付物：

- 第一批高质量条文
- 第一批高质量来源 JSON
- 第一版关键词标准层

### 阶段 4：建立 SQLite 运行层

这一阶段是技术底座阶段。

目标是：

`不改变源数据维护方式，但让运行、查询、统计、审计更稳。`

必须完成的事项：

1. 建立数据库 schema
2. 建立条文导入脚本
3. 建立来源导入脚本
4. 建立关键词导入与去重逻辑
5. 建立关系表写入逻辑

这一阶段交付物：

- `storage/app.db`
- 导入脚本
- 可重复执行的入库流程

### 阶段 5：建立审计和辅助能力

这一阶段的目标不是“自动接管”，而是“减少人工重复劳动”。

建议完成：

1. 数据审计脚本
2. 弱关键词警告
3. 条文关键词密度警告
4. 来源片段长度警告
5. TXT 切段草稿生成
6. 候选关键词推荐机制

这一阶段交付物：

- 审计脚本
- 审计报告
- 半自动整理工具

## 七、条文层执行清单

条文层是关系图的起点，必须优先整理。

### 7.1 编号统一

建议目标：

- 内部编号全局唯一
- 展示编号允许保留原书条号

建议格式：

- `shanghan-265`
- `jingui-014`
- `wenbing-023`

建议执行项：

1. 盘点现有条文编号
2. 找出重复编号
3. 找出不规则编号
4. 决定是否保留 `display_no`
5. 决定数据库主键使用内部编号

### 7.2 条文字段统一

每条条文至少应有：

1. `id`
2. `title`
3. `content`
4. `translation`
5. `keywords`

建议补充：

1. `book_id`
2. `chapter_id`
3. `display_no`
4. `source_file`
5. `status`

### 7.3 条文状态分层

建议执行项：

1. 增加条文状态字段
2. 区分草稿条文和已检查条文
3. 区分关键词已检查和关系已检查

状态建议：

- `draft`
- `keyword_checked`
- `relation_checked`
- `published`

### 7.4 条文优先级

后面补数据不要平均补。

建议优先顺序：

1. 当前首页和演示中出现的经典
2. 用户已经频繁使用的条文
3. 已有关键词但关系结果差的条文
4. 没有关键词的核心条文
5. 边缘经典

---

## 八、关键词层执行清单

关键词层是后面最有价值的数据资产。

### 8.1 建立关键词分类

建议至少分 5 类：

1. 症状词
2. 脉象词
3. 病机词
4. 部位词
5. 方药词

后面可扩展：

1. 病名词
2. 治法词
3. 情志词
4. 体质词

### 8.2 建立关键词质量等级

建议等级：

- `A`：高质量关键词
- `B`：可用关键词
- `C`：存疑关键词
- `D`：禁用或泛词

示例：

- `谵语`：A
- `少阳`：A
- `胃不和`：A
- `热`：D
- `病`：D

### 8.3 建立关键词别名层

后面不建议简单粗暴把所有近似词直接合并。

建议建立别名表。

建议执行项：

1. 盘点高频变体词
2. 标准词和别名词分开存
3. 前台展示保留原样
4. 后台匹配可走标准化映射

### 8.4 关键词长度和密度控制

建议规则：

- 小于 2 个字：警告
- 超过 12 个字：警告
- 超过 30 个字：禁止

每条条文建议：

- 推荐 4 到 8 个关键词
- 上限 12 个
- 超过 12 个进入“过密”警告

---

## 九、来源片段层执行清单

右侧命中体验，决定用户对产品的第一印象。

### 9.1 来源片段必须片段化

不要把整本来源当成一条命中结果。

建议目标长度：

- 30 到 200 字最佳
- 超过 300 字建议切段
- 少于 10 字通常信息不足

### 9.2 来源片段的最小字段

建议至少有：

1. `source_name`
2. `category`
3. `title`
4. `content`
5. `keywords`

### 9.3 来源层级要分清

建议后台分 3 层：

1. 来源书目
2. 来源篇名
3. 来源片段

前端主要展示第 3 层，但后台要保留前两层，便于追溯和分组。

### 9.4 JSON 优先，TXT 回退继续保留

这个规则当前已经很好，不建议推翻。

但建议把 TXT（文本文件）命中结果明确标记出来。

建议执行项：

1. 为 TXT 命中标记 `match_type`
2. 为 TXT 命中增加低置信度提示
3. 优先把高频 TXT 来源转成 JSON

## 十、SQLite 运行层执行清单

这部分的原则是：

`并行接入，不要一次性强制切换。`

### 10.1 建议表结构

建议优先建立：

1. `books`
2. `chapters`
3. `clauses`
4. `keywords`
5. `clause_keywords`
6. `relation_sources`
7. `relation_entries`
8. `relation_entry_keywords`

### 10.2 第一批入库目标

建议不要一上来全量导入全部来源。

先选：

1. 《伤寒论》
2. 《金匮要略》
3. 《温病条辨》
4. 1 到 3 个高频关联来源

### 10.3 数据库接入顺序

建议顺序：

1. 先建库
2. 再写导入脚本
3. 再做读接口
4. 再切写接口
5. 最后才切前端主读取

### 10.4 最适合优先切数据库的点

建议优先切：

1. 添加关键词
2. 删除关键词

因为这两类操作最明确，也最容易验证。

## 十一、数据审计脚本执行清单

这一部分强烈建议尽快做。

没有它，后面补数据会很慢。

建议审计至少输出下面这些结果。

### 11.1 没有关键词的条文

输出内容：

- 条文编号
- 书名
- 标题
- 原文长度

### 11.2 有关键词但零命中的条文

输出内容：

- 条文编号
- 标题
- 关键词列表
- 当前命中数

### 11.3 弱关键词和泛关键词

输出内容：

- 关键词
- 出现次数
- 推荐等级
- 是否建议禁用

### 11.4 来源登记异常

输出内容：

- 已登记但文件缺失
- 文件存在但未登记
- 同名冲突来源

### 11.5 重复来源片段

输出内容：

- 来源名
- 篇名
- 相同内容次数

### 11.6 只有 TXT 没有 JSON 的来源

输出内容：

- 来源名
- 分类
- 是否高频命中

## 十五、最后的结论

第二阶段的数据层整理，不是去“发明更多节点”，而是把当前已经确定的三类节点做成一个稳定体系。

后面应该围绕下面这件事持续推进：

`让条文更干净，让关键词更稳定，让来源片段更可读，让所有关系都能追溯。`

如果后面继续按这个方向走，这个项目的数据层会越来越稳，后面不管是继续补数据、切数据库，还是做自动辅助整理，都会容易很多。
