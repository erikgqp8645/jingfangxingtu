# 经方星图新手数据维护手册

这份手册是给第一次接触这个项目的人写的。

目标只有一个：

让你不需要懂编程，也能按步骤维护这个项目的数据。

如果你愿意，你可以把这份手册理解成“保姆级说明书”。

---

## 1. 先用一句话理解项目

这个项目会做下面这件事：

`拿经典条文里的关键词，去关联解析资料里找相同关键词，然后把它们连起来。`

比如：

- 经典条文里有关键词 `谵语`
- 关联解析资料里也有 `谵语`
- 系统就会把这两边连起来

所以，这个项目最核心的数据其实只有两大块：

1. 经典条文数据
2. 关联解析数据

---

## 2. 你平时最常改的地方

你最常接触的目录是：

```text
data/
├─ jingdianconfig.json
├─ guanlianjiexiconfig.json
├─ 经典/
├─ 关联解析/
└─ 白话解析/
```

下面是它们的作用。

### 2.1 `data/jingdianconfig.json`

作用：

告诉系统“有哪些经典、每本经典有哪些章节、每章有哪些条文”。

你可以把它理解成“目录总表”。

### 2.2 `data/经典/`

作用：

存放每一条经典条文的详细内容。

每条条文一般对应一个 JSON 文件。

### 2.3 `data/guanlianjiexiconfig.json`

作用：

告诉系统“有哪些关联解析来源可以用来匹配关键词”。

### 2.4 `data/关联解析/`

作用：

存放关联解析资料。

这里的资料可以是：

- JSON
- TXT

### 2.5 `storage/app.db`

作用：

这是数据库文件。

但请记住：

`平时先改 data，再导入数据库。`

不要把数据库当成你第一个动手编辑的地方。

---

## 3. 最重要的原则

请反复记住下面这 4 句话。

### 原则 1：先改 `data/`

因为 `data/` 最容易看懂，也最容易改。

### 原则 2：数据库是“后导入”的，不是“先手改”的

数据库主要是为了让项目跑得更顺。

### 原则 3：先保证文件格式正确，再考虑导入数据库

很多问题不是业务逻辑错，而是 JSON 格式错了。

### 原则 4：每次只改一小步，改完就刷新看效果

这样最安全。

---

## 4. 从零开始的最小操作流程

如果你今天要维护数据，请照这个顺序来。

### 情况 A：只是给已有条文加关键词

最简单。

顺序：

1. 打开页面
2. 找到条文
3. 在正文区选中文字
4. 右键添加关键词
5. 刷新看右侧关联有没有变化

这时候系统会自动同步：

- SQLite
- 对应条文 JSON

### 情况 B：手工修改某条条文 JSON

顺序：

1. 打开对应的条文 JSON
2. 修改 `content`、`translation` 或 `keywords`
3. 保存文件
4. 刷新页面
5. 检查显示是否正确

### 情况 C：新增一个关联解析来源

顺序：

1. 把 TXT 或 JSON 放到 `data/关联解析/`
2. 在 `data/guanlianjiexiconfig.json` 里登记它
3. 刷新页面检查效果
4. 如需数据库同步，再执行关系导入命令

### 情况 D：新增一本经典

顺序：

1. 先准备条文 JSON 文件
2. 再修改 `data/jingdianconfig.json`
3. 刷新页面检查目录是否出来
4. 再决定是否导入数据库

这是正确顺序。

不是先建数据库。

## 6. 如何添加关键词

这个项目现在已经支持“页面里直接添加关键词”。

### 方法 1：页面右键添加

这是最推荐的方法。

操作步骤：

1. 打开页面
2. 在左侧选择一本经典
3. 再选择一条条文
4. 在中间“正文原文区”用鼠标选中你想当关键词的文字
5. 点击鼠标右键
6. 在弹出的框里点击“添加为关键词”

例如：

原文是：

```text
少阳不可发汗，发汗则谵语，此属胃，胃和则愈；胃不和，烦而悸。
```

你可以选中：

- 少阳
- 谵语
- 胃不和
- 烦而悸

然后右键添加。

### 方法 2：直接改 JSON

打开条文 JSON，找到 `keywords`。

例如：

```json
{
  "id": "265",
  "title": "伤寒论 第265条",
  "content": "伤寒，脉弦细，头痛发热者，属少阳。少阳不可发汗，发汗则谵语，此属胃，胃和则愈；胃不和，烦而悸。",
  "translation": "这是条文的白话解释。",
  "keywords": ["脉弦细", "头痛发热", "少阳", "谵语", "胃不和", "烦而悸"]
}
```

如果你要加一个新关键词，比如 `伤寒`，改成：

```json
{
  "id": "265",
  "title": "伤寒论 第265条",
  "content": "伤寒，脉弦细，头痛发热者，属少阳。少阳不可发汗，发汗则谵语，此属胃，胃和则愈；胃不和，烦而悸。",
  "translation": "这是条文的白话解释。",
  "keywords": ["伤寒", "脉弦细", "头痛发热", "少阳", "谵语", "胃不和", "烦而悸"]
}
```

### 添加关键词时要注意什么

1. 关键词不要太长
2. 关键词最好是清楚的词，不要太泛
3. 关键词尽量和关联解析里的词一致

不推荐的词：

- 病
- 热
- 寒

推荐的词：

- 谵语
- 少阳
- 脉弦细
- 胃不和

## 7. 如何删除关键词

现在支持“在关键词列表里删除”。

操作步骤：

1. 打开一条条文
2. 在关键词区域找到要删的词
3. 点击该关键词右边的小“删”按钮
4. 在弹出的确认框里点“确定”

删除后会同时发生这几件事：

1. 当前条文的关键词会被删掉
2. JSON 会同步更新
3. 数据库会同步更新
4. 右侧关联命中会重新计算
5. 中间关系图也会重新变化

要特别注意：

删除的是“这条条文上的这个关键词关联”。

不是把全项目所有地方的这个词都删掉。

例如：

- 条文 A 有 `谵语`
- 条文 B 也有 `谵语`

你在条文 A 里删掉 `谵语`，不会影响条文 B。

## 8. 经典条文 JSON 怎么写

这是最重要的一部分。

每条经典条文一般是一个 JSON 文件。

最常见格式如下：

```json
{
  "id": "265",
  "title": "伤寒论 第265条",
  "content": "伤寒，脉弦细，头痛发热者，属少阳。少阳不可发汗，发汗则谵语，此属胃，胃和则愈；胃不和，烦而悸。",
  "translation": "外感伤寒，如果出现脉弦细、头痛发热，就属于少阳病。少阳病不能用发汗法，如果误发汗，可能出现谵语。这和胃气有关，胃气调和就会好；胃气不和，就会烦躁心悸。",
  "keywords": ["伤寒", "脉弦细", "头痛发热", "少阳", "谵语", "胃不和", "烦而悸"]
}
```

### 每个字段是什么意思

#### `id`

条文的唯一编号。

不能和别的条文重复。

例子：

- `265`
- `jingui-06-14`
- `wenbing-01-01`

#### `title`

条文标题。

就是页面上看到的名字。

#### `content`

原文正文。

也就是经典条文原文。

#### `translation`

白话解释。

可以为空字符串，但建议慢慢补齐。

#### `keywords`

关键词数组。

系统就是靠它去匹配关联解析的。

## 9. `jingdianconfig.json` 怎么写

这个文件决定左侧目录怎么显示。

示例：

```json
[
  {
    "id": "shanghan",
    "name": "《伤寒论》",
    "chapters": [
      {
        "title": "辨少阳病脉证并治",
        "clauses": [
          {
            "id": "265",
            "title": "第265条 · 少阳误汗变证",
            "dataFile": "/data/经典/伤寒论/265.json"
          }
        ]
      }
    ]
  }
]
```

### 你只要记住三层

1. 书
2. 章节
3. 条文

### 关键字段

#### 书这一层

- `id`
- `name`
- `chapters`

#### 章节这一层

- `title`
- `clauses`

#### 条文这一层

- `id`
- `title`
- `dataFile`

### `dataFile` 是什么

它就是指向条文 JSON 的路径。

例如：

```text
/data/经典/伤寒论/265.json
```

这表示真实文件在：

[data/经典/伤寒论/265.json](C:\Users\hxst01\Documents\aicoding\jingfangxingtu\data\经典\伤寒论\265.json)

## 10. 如何新增一条经典条文

这里给你一个最简单、最安全的顺序。

### 第一步：先创建条文 JSON 文件

比如你要新增：

`第999条`

那你先在：

[data/经典/伤寒论/999.json](C:\Users\hxst01\Documents\aicoding\jingfangxingtu\data\经典\伤寒论\999.json)

写入：

```json
{
  "id": "999",
  "title": "伤寒论 第999条",
  "content": "这里写原文。",
  "translation": "这里写白话解释。",
  "keywords": ["这里", "写", "关键词"]
}
```

### 第二步：再去 `jingdianconfig.json` 里登记

在对应章节 `clauses` 里加一条：

```json
{
  "id": "999",
  "title": "第999条 · 这里写条文标题",
  "dataFile": "/data/经典/伤寒论/999.json"
}
```

### 第三步：保存后刷新页面

如果路径和 JSON 格式都对，左侧目录就会出现新条文。

### 第四步：需要数据库时再导入

如果你希望数据库里也同步，就执行：

```bash
npm run db:import:shanghan
```

先看左侧有没有这本书。

不是所有新增经典都必须马上写数据库导入脚本。

如果你只是测试显示，先保证 JSON 流程通了再说。

## 12. 关联解析文件怎么加

### 12.1 先理解一件事

关联解析是“被搜索的一方”。

经典条文有关键词。

系统会去关联解析资料里找相同的词。

## 13. `guanlianjiexiconfig.json` 怎么写

这个文件用来登记有哪些关联解析来源。

示例：

```json
[
  {
    "sourceName": "黄帝内经素问",
    "fileBaseName": "437-黄帝内经素问",
    "category": "内经"
  },
  {
    "sourceName": "伤寒寻源",
    "fileBaseName": "467-伤寒寻源",
    "category": "伤寒"
  }
]
```

### 字段解释

#### `sourceName`

页面显示用的名称。

#### `fileBaseName`

真正关联到文件名的“主名字”。

例如：

```text
437-黄帝内经素问.json
437-黄帝内经素问.txt
```

它们的 `fileBaseName` 都是：

```text
437-黄帝内经素问
```

#### `category`

分类名。

主要用于显示和分组。

## 14. 关联解析 JSON 怎么写

这是最推荐的格式。

因为最准。

示例：

```json
{
  "谵语": [
    {
      "title": "病能论篇第四十六",
      "content": "其病发热、汗出、谵语。谵语者，阳气独盛，五脏气之所不营也，故心神狂越而妄言。"
    }
  ],
  "恶寒": [
    {
      "title": "宣明五气篇第二十三",
      "content": "五脏所恶：心恶热，肺恶寒，肝恶风，脾恶湿，肾恶燥，是谓五恶。"
    }
  ]
}
```

### 这个格式怎么理解

外层的键，就是关键词。

比如：

- `谵语`
- `恶寒`

每个关键词后面是一个数组。

数组里可以放一条，也可以放多条。

每一条至少要有：

- `title`
- `content`

### 什么时候适合用 JSON

如果你已经知道某个关键词对应哪段解释，最适合用 JSON。

因为它是“直接命中”。

## 15. 关联解析 TXT 怎么写

TXT 适合放原始资料全文。

例如：

```text
<目录>卷第七

<篇名>宣明五气篇第二十三

属性：五味所入∶酸入肝，辛入肺，苦入心，咸入肾，甘入脾，是谓五入。
五气所病∶心为噫，肺为咳，肝为语，脾为吞，肾为欠为嚏……
五脏所恶∶心恶热，肺恶寒，肝恶风，脾恶湿，肾恶燥，是谓五恶。
```

系统会做什么：

1. 按 `<篇名>` 或类似标记切分
2. 查找关键词
3. 找到后截取附近片段显示到右侧

### TXT 什么时候适合用

适合下面这两种情况：

1. 你手里只有原文全文
2. 你暂时没空把它整理成 JSON

### 重要规则

如果同一个来源同时有 JSON 和 TXT：

1. 先查 JSON
2. 某个关键词 JSON 没命中，再回退到 TXT

这叫：

`关键词级别回退`

不是整本书只选一个格式。

## 16. 经典条文和关联解析怎么对应

看一个完整例子。

### 经典条文

```json
{
  "id": "1",
  "title": "伤寒论 第1条",
  "content": "太阳之为病，脉浮，头项强痛而恶寒。",
  "translation": "太阳病的特点，是脉浮、头项强痛，而且怕冷。",
  "keywords": ["太阳", "脉浮", "头项强痛", "恶寒"]
}
```

### 关联解析 JSON

```json
{
  "恶寒": [
    {
      "title": "宣明五气篇第二十三",
      "content": "五脏所恶：心恶热，肺恶寒，肝恶风，脾恶湿，肾恶燥，是谓五恶。"
    }
  ]
}
```

系统会做什么：

1. 读到条文关键词 `恶寒`
2. 去 `data/关联解析/` 里找配置好的来源
3. 如果某个来源的 JSON 里正好有 `恶寒`
4. 就把这条内容显示到右侧
5. 同时在图里形成关联

## 17. 什么时候需要导入数据库

这是很多新手最容易迷糊的地方。

最简单的判断方式如下。

### 不一定需要马上导数据库的情况

- 你只是改了一点 JSON
- 你只是改了 TXT
- 你只是想在开发环境里先看效果

因为当前项目很多地方已经支持直接从文件读取或回退。

### 建议导数据库的情况

- 你批量新增了很多经典条文
- 你想让接口统一走数据库
- 你准备长期稳定使用这批数据
- 你想避免某些数据只停留在文件模式

---

## 18. 数据库导入顺序应该是什么

这里给你最稳的顺序。

### 18.1 全新初始化时

先：

```bash
npm run db:init
```

然后按经典顺序导入：

```bash
npm run db:import:shanghan
npm run db:import:jingui
npm run db:import:wenbing
```

再导入关联解析：

```bash
npm run db:import:relations
```

### 18.2 一步完成

如果你就是想全部重建，直接：

```bash
npm run db:seed
```

### 18.3 为什么要这样排

因为逻辑顺序就是：

1. 先有经典
2. 再有条文
3. 再有关键词
4. 最后再让关联解析去和这些关键词对应

虽然某些时候顺序不严格也能跑，但新手最好就按这个顺序来。

## 19. 每次加数据时的推荐顺序

顺序：

1. 先建经典目录
2. 再准备条文 JSON
3. 再改 `jingdianconfig.json`
4. 刷新检查
5. 最后再考虑数据库脚本

## 20. 常见错误

下面这些错误非常常见。

### 错误 1：先改数据库，不改 `data/`

不推荐。

因为以后别人不知道源头在哪。

### 错误 2：`jingdianconfig.json` 写了，但条文 JSON 路径不对

结果：

左侧目录可能有，点进去内容是空的。

### 错误 3：JSON 少逗号、多逗号、少引号

结果：

页面加载失败，或者接口报错。

### 错误 4：关键词写得太泛

比如：

- 热
- 寒
- 病

结果：

命中太多，关系图会很乱。

### 错误 5：关联解析来源忘记登记到 `guanlianjiexiconfig.json`

结果：

文件明明放进去了，但系统根本不会查它。

### 错误 6：只放了 TXT，没有结构分段

结果：

系统虽然能查，但右侧显示不够精确。

## 22. 推荐工作习惯

最后给你一个很实用的习惯。

1. 先改一小步
2. 先保存
3. 先刷新页面看
4. 没问题再改下一步
5. 一轮都确认完，再导数据库

这个节奏最稳。
