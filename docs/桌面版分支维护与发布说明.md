# 桌面版分支维护与发布说明

这份说明专门讲一件事：

`Windows 桌面版后续只在独立分支维护，不直接把开发中的桌面版改动并入 main。`

---

## 1. 当前约定

当前桌面版长期维护分支是：

- `codex/windows-desktop-app`

从现在开始，按下面的规则执行：

1. 桌面版新功能、打包、安装器、桌面维护文档，都优先在 `codex/windows-desktop-app` 上开发
2. 不主动把桌面版开发中的改动推到 `main`
3. 只有在你明确决定同步主线时，才处理和 `main` 的合并关系
4. 如果 `main` 后续有通用功能更新，再评估要不要同步到桌面版分支

---

## 2. 为什么这样维护

这样做有几个实际好处：

- 避免桌面版实验性改动影响主线
- 浏览器版和桌面版的节奏可以分开
- 桌面打包、安装、升级相关改动比较集中，后续排查更容易
- 以后如果要长期做 Windows 应用，这个分支可以直接当桌面产品线

---

## 3. 什么时候才考虑和 `main` 交互

只有下面几种情况，才建议处理 `main`：

- `main` 修了一个桌面版也需要的通用 bug
- `main` 新增了桌面版也要用的数据结构
- 你准备正式收敛桌面版架构，想把一部分稳定能力回灌主线

其余情况下，优先保持桌面版分支独立。

---

## 4. 推荐工作流

### 4.1 日常开发

在桌面版分支上工作：

```bash
git checkout codex/windows-desktop-app
```

开发和验证常用命令：

```bash
npm run desktop:dev
npm run build:desktop:dir
npm run build:desktop
```

### 4.2 当 `main` 有变化时

先看 `main` 的变化是否真的和桌面版相关。

如果相关，再决定：

- 合并 `main` 到桌面版分支
- 或者只挑需要的提交过来

不要因为 `main` 变了，就机械地同步一遍。

---

## 5. GitHub Release 的建议做法

桌面版如果继续独立维护，推荐也按“分支版本”发布：

- 分支：`codex/windows-desktop-app`
- 版本标签：例如 `desktop-v0.1.0`
- Release 页面内容：写清楚这是桌面版，不是主线浏览器版

这样以后你一眼就能分清：

- 哪些是桌面版发布
- 哪些是主线版本

---

## 6. 当前桌面版已具备的发布能力

当前桌面版已经具备这些能力：

- Electron 桌面壳
- 本地服务独立化
- Windows 安装包构建
- 首次启动复制可写数据到用户目录
- 桌面图标和安装包基础元数据
- 桌面维护手册

对应文档：

- `docs/桌面版打包与维护手册.md`

---

## 7. 给后续维护者的一句话

如果以后你重新接手这个版本，请优先记住：

`桌面版是一个独立产品线，优先维护 codex/windows-desktop-app，不要默认直接动 main。`
