# 第二阶段执行计划

## 一、阶段名称

第二阶段：数据闭环与规模化整理

---

## 二、这一阶段的核心目标

这一阶段不再以“继续加很多新功能”为主。

这一阶段的重点是把现在已经能跑起来的系统，变成一个：

- 数据维护有闭环
- 数据同步不混乱
- 数据补充能持续推进
- 桌面版后续容易维护

的项目。

简单说，这一阶段主要解决两个问题：

1. 改了数据之后，怎么确保前端、数据库、桌面版都一致。
2. 现在条文很多，但关键词和来源还太少，怎么高效率往前补。

---

## 三、当前真实情况

根据当前体检结果，项目现在的情况大致是：

- 条文总数：1265
- 还没有关键词的条文：1189
- 已有关键词的条文：76
- 零命中条文：0
- 零命中关键词：0
- 关联来源数：3
- JSON 命中总数：222
- TXT 命中总数：4070
- 长度异常片段：212

这说明：

1. 现在系统主体已经通了。
2. 当前最大问题不是代码跑不起来。
3. 当前最大问题是数据覆盖率太低。
4. 现在的命中还非常依赖 TXT，不够结构化。

所以第二阶段的方向是对的：

- 先稳住数据闭环
- 再持续补关键词
- 再提升关系来源质量

---

## 四、第二阶段只做 3 条主线

这一阶段建议只抓下面 3 条主线，不要再分散。

### 主线 A：数据闭环

目标：

- 改完 JSON 后，能明确知道有没有同步到 SQLite
- 前端看到的数据和数据库里的数据保持一致
- 桌面版也遵守同一套规则

### 主线 B：批量补数据

目标：

- 连续补《金匮要略》后续章节
- 把“没有关键词的条文”数量稳定往下压
- 保持“零命中条文 = 0”

### 主线 C：提升关系来源质量

目标：

- 不只靠 TXT 大段模糊命中
- 把高频命中的 TXT 逐步整理成更短、更准的 JSON 片段
- 为以后数据库检索统一做准备

---

## 五、主线 A：数据闭环执行计划

### A-1 先确定一条铁规则

这条规则以后不要摇摆：

`平时先改 data 目录，再执行 npm run db:sync 同步到 SQLite。`

也就是说：

- `data/` 是维护入口
- `SQLite` 是运行时主读取源
- `db:sync` 是两者之间的同步桥梁

### A-2 要补一个“同步状态可见”

现在虽然已经有 `npm run db:sync`，但还不够“看得见”。

建议补一个很小的机制，让用户一眼知道：

- 当前数据库是不是最新
- 最近一次同步是什么时候
- 当前数据是否可能还是旧的

建议实现方式：

1. 在数据库或本地状态文件里记录最后同步时间
2. 前端页脚或设置区显示：
   - 最近同步时间
   - 当前数据库版本状态
3. 同步完成后给出明确提示

### A-3 需要补一个“同步后检查”

每次同步完成后，最少自动检查下面 3 项：

1. `jingui-01-1` 这类已知样本条文，关键词是否进入数据库
2. 数据库条文数是否和配置条文数一致
3. 关系来源数是否和 `guanlianjiexiconfig.json` 一致

### A-4 桌面版也要遵守这套闭环

桌面版后面如果继续维护数据，也必须遵守同一规则：

1. 维护用户目录里的 `data/`
2. 运行同步
3. 再打开界面检查

### A 主线验收标准

做到下面几点，就算 A 主线完成：

- 能看到最近一次同步时间
- 能清楚区分“改了 JSON 但还没同步”与“已经同步”
- 前端条文关键词和数据库内容一致
- 桌面版和网页版使用同一套同步规则

## 六、主线 B：批量补数据执行计划

### B-1 补数据的顺序不要乱

建议按下面顺序走：

1. 先补《金匮要略》第 6 章
2. 再补第 7 章
3. 再补第 8 章
4. 再补第 9 章
5. 再补第 10 章
6. 之后再回头补特别长、特别难的条文

原因：

- 这样能连续推进
- 同一章的术语更容易复用
- 更容易同步补关系来源

### B-2 每条条文都按同一规则补

每补一条，建议固定做这几步：

1. 读原文
2. 提取 3 到 8 个最贴原文的关键词
3. 写入条文 JSON
4. 检查这些关键词有没有来源命中
5. 如果没有，就补到关联来源 JSON/TXT
6. 运行 `npm run db:sync`
7. 运行 `npm run data:audit`

### B-3 关键词提取原则

优先级建议如下：

1. 病名
2. 脉象
3. 证候
4. 方名
5. 核心治法
6. 关键症状短语

尽量不要优先补下面这些：

- 太泛的词
- 单个虚词
- 只有上下文里才有意义的普通字词

### B-4 每一批的规模建议

不要一次补太多。

建议每一批控制在：

- 5 到 15 条条文

原因：

- 容易核对
- 容易同步到数据库
- 容易排查问题

### B-5 每一批都要看 4 个数字

每次补完后，重点看下面 4 个数字：

1. 没有关键词的条文是否下降
2. 零命中条文是不是还是 0
3. 零命中关键词是不是还是 0
4. JSON 命中总数有没有提升

### B 主线验收标准

做到下面几点，就算 B 主线完成得好：

- 《金匮要略》第 6 到第 10 章完成一轮关键词补充
- “没有关键词的条文”数量持续下降
- 保持“零命中条文 = 0”
- 保持“零命中关键词 = 0”

---

## 七、主线 C：关系来源质量执行计划

### C-1 先不要急着加很多新来源

当前来源只有 3 个，确实少。

但第二阶段前半段，不建议一下子接很多书。

更建议先做两件事：

1. 把现有高频来源整理得更结构化
2. 再挑 1 到 2 个高价值新来源接入

### C-2 先处理超长 TXT 片段

当前体检里已经看见：

- 长度异常片段：212

这类片段的问题是：

- 命中很多
- 但不够准
- 用户看起来会觉得太长、太乱

建议处理顺序：

1. 先看高频命中来源
2. 找出最长、最常命中的 TXT 段
3. 用拆分脚本生成草稿
4. 再人工确认后写成 JSON

### C-3 关系来源 JSON 的整理原则

每个 JSON 片段尽量做到：

- 一条关键词对应一个清楚片段
- 片段尽量短
- 标题清楚
- 不要过度重复

### C-4 新来源接入建议

第二阶段后半段，可以新增 1 到 2 个来源。

选择标准：

- 和《金匮要略》《伤寒论》高频术语重合多
- 原始文本质量较好
- 容易拆成片段

### C 主线验收标准

做到下面几点，就算 C 主线有成果：

- 长片段数量开始下降
- JSON 片段数持续增加
- 右侧命中内容更短、更准
- 再新增 1 到 2 个高价值来源

## 八、第二阶段建议的具体开发顺序

## 九、这一阶段不要做什么

为了避免分散精力，这一阶段不建议优先做下面这些：

- 不要优先做复杂 AI 自动补词
- 不要优先做很复杂的图谱算法
- 不要优先做大量视觉改版
- 不要优先接特别多新经典
- 不要优先做复杂权限系统

原因很简单：

这些都不是当前最大瓶颈。

当前最大瓶颈还是：

- 数据不够多
- 数据不够稳
- 数据闭环还不够可见

---

## 十、阶段完成标志

如果第二阶段做得比较扎实，最后应该看到这样的结果：

- 同步逻辑对你来说已经不混乱
- 前端、数据库、桌面版的数据关系清楚
- 《金匮要略》大块数据已经被连续补完
- 关系来源不再严重依赖超长 TXT
- 项目从“能演示”进入“能持续维护”

---

## 十一、下一次开工时，建议直接从这里开始

如果下一次继续开发，建议直接执行这条顺序：

1. 先做“同步状态可见”
2. 再从《金匮要略》第 6 章开始继续补
3. 每补一批就同步数据库并跑体检
4. 再开始整理高频 TXT 片段

如果你不想重新思考，只想直接往下做，就按这 4 步继续。
