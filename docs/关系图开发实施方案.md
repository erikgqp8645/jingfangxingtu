# 经方关系图开发实施方案

## 1. 目标

基于 [关系图需求说明.md](/C:/Users/hxst01/Documents/aicoding/jingfangxingtu/docs/关系图需求说明.md)，将当前项目从“知识图谱展示”调整为“关键词驱动的关系图展示”。

本方案聚焦以下三个结果：

- 降低条文数据建模复杂度
- 保留跨文本关联检索能力
- 将前端展示从复杂图谱改为可解释的关系图

## 2. 当前项目现状

### 2.1 当前数据结构

当前项目存在两套混合结构：

- `data/jingdianconfig.json`：经典目录配置
- `src/data.ts`：内置静态条文对象，包含 `nodes`、`links`、`plugins`

当前条文数据结构偏向知识图谱模型，例如：

- `nodes`
- `links`
- `plugins.bingyuan / neijing / mingli / benjing / tangye`

这导致条文本身需要承担“展示节点定义”和“语义关系建模”的职责，维护成本较高。

### 2.2 当前检索逻辑

当前检索逻辑位于 [searchUtils.ts](/C:/Users/hxst01/Documents/aicoding/jingfangxingtu/src/lib/searchUtils.ts)，已经具备以下能力：

- 预加载 `guanlianjiexiconfig.json`
- 优先读取 `关联解析/*.json`
- 回退读取 `关联解析/*.txt`
- 根据关键词返回结构化或文本型命中结果

这一部分与目标需求方向基本一致，可以保留并重构。

### 2.3 当前页面结构

当前页面主结构位于 [App.tsx](/C:/Users/hxst01/Documents/aicoding/jingfangxingtu/src/App.tsx)：

- 左侧 `Sidebar`：经典与条文导航
- 中间 `GraphView + ClauseDetail`：图谱与条文详情
- 右侧 `PluginPanel`：关联解析展示

其中需要重点调整的是：

- `GraphView` 的数据输入
- `PluginPanel` 的展示依据
- `src/data.ts` 的静态知识图谱定义

## 3. 改造原则

### 3.1 保留已有可用部分

优先复用以下现有能力：

- `data/` 作为单一数据源
- Vite 将 `data/` 映射为 `/data/*`
- 关联解析配置与文件组织方式
- `searchKnowledgeBase` 的 JSON/TXT 双通道检索思路

### 3.2 删除或弱化重建模部分

应逐步弱化以下内容：

- 条文级 `nodes`
- 条文级 `links`
- 手工维护的大量知识图谱节点
- 以插件语义分类为核心的前端展示逻辑

### 3.3 采用最小可行关系模型

前端和数据层统一围绕以下三类实体组织：

- `Clause`：经典条文
- `Keyword`：条文关键词
- `RelationHit`：关联解析命中片段

## 4. 目标数据模型

### 4.1 Clause

建议定义新的条文接口：

```ts
export interface ClauseData {
  id: string;
  title: string;
  content: string;
  translation: string;
  keywords: string[];
}
```

说明：

- 删除 `nodes`
- 删除 `links`
- 删除 `plugins`
- 保留最小展示与检索字段

### 4.2 RelationHit

建议新增命中结果结构：

```ts
export interface RelationHit {
  id: string;
  keyword: string;
  sourceName: string;
  category: string;
  title: string;
  content: string;
  matchType: 'json' | 'txt';
}
```

说明：

- `keyword` 用于解释命中依据
- `sourceName` 用于显示来源
- `category` 用于分组
- `matchType` 用于标记是 JSON 命中还是 TXT 命中

### 4.3 RelationGraph

建议关系图组件使用扁平输入：

```ts
export interface RelationNode {
  id: string;
  label: string;
  type: 'clause' | 'keyword' | 'source';
}

export interface RelationLink {
  source: string;
  target: string;
}
```

关系规则：

- `clause -> keyword`
- `keyword -> source`

如需更细粒度，也可将 `source` 替换为“文献片段节点”。

## 5. 数据层改造方案

### 5.1 经典配置文件

保留 [jingdianconfig.json](/C:/Users/hxst01/Documents/aicoding/jingfangxingtu/data/jingdianconfig.json) 作为经典目录入口，但应修改其 `data` 字段用途。

当前模式：

- `data` 可能是 `static_clause265`
- 或 `dynamic_29`

建议改为：

- `dataFile`：指向条文详情 JSON
- 或直接内嵌最小条文结构

推荐方案：

```json

  "title": "第 29 条 · 太阳误攻其表",
  "dataFile": "/data/经典/伤寒论/29.json"

```

理由：

- 条文详情数据脱离前端源码
- 非开发人员可以直接维护
- 逐步摆脱 `src/data.ts`

### 5.2 条文详情文件

建议新增独立条文 JSON 文件，存放于 `data/经典/` 下。

建议结构：

```json
{
  "id": "29",
  "title": "伤寒论 第29条",
  "content": "伤寒脉浮、自汗出……",
  "translation": "太阳病误治后……",
  "keywords": ["谵语", "吐逆"]
}
```

### 5.3 关联解析配置

保留 [guanlianjiexiconfig.json](/C:/Users/hxst01/Documents/aicoding/jingfangxingtu/data/guanlianjiexiconfig.json)。

建议新增可选字段：

- `priority`
- `enabled`
- `displayName`

示例：

```json
{
  "sourceName": "黄帝内经素问",
  "displayName": "《黄帝内经素问》",
  "fileBaseName": "437-黄帝内经素问",
  "category": "内经",
  "enabled": true,
  "priority": 1
}
```

### 5.4 关联解析 JSON 文件

应统一采用“关键词索引”结构，避免每个文件各自定义格式。

统一要求：

- 顶层键为关键词
- 值为数组
- 数组元素至少包含 `title` 和 `content`

### 5.5 关联解析 TXT 文件

TXT 文件不强制改格式，但解析器应统一：

- 按篇名或标题切分
- 命中关键词后返回有限长度摘要
- 输出结构映射为 `RelationHit`

## 6. 前端改造方案

### 6.1 App 组件

`App.tsx` 需要从“取条文图谱数据并扩展图谱”改为“取条文并生成关系图”。

当前主要问题：

- `getExpandedGraph` 仍基于 `nodes` 和 `links`
- 条文对象中混入图谱结构

改造后应变为：

1. 加载经典目录
2. 解析当前条文详情
3. 读取条文 `keywords`
4. 调用关系检索函数获取命中结果
5. 用条文和命中结果生成关系图数据

### 6.2 GraphView 组件

建议将 [GraphView.tsx](/C:/Users/hxst01/Documents/aicoding/jingfangxingtu/src/components/GraphView.tsx) 重命名或重构为 `RelationGraphView.tsx`。

改造目标：

- 节点类型缩减为 `clause / keyword / source`
- 颜色和布局按关系图而非知识图谱设计
- 避免出现过多自动扩展节点
- 图上强调“命中来源”而不是“病机推导”

### 6.3 PluginPanel 组件

当前 [PluginPanel.tsx](/C:/Users/hxst01/Documents/aicoding/jingfangxingtu/src/components/PluginPanel.tsx) 主要依赖条文里的 `plugins` 字段，并混合动态搜索结果。

建议改为纯检索结果面板：

- 输入为 `RelationHit[]`
- 按 `category` 或 `sourceName` 分组
- 每条结果展示：
  - 命中关键词
  - 来源
  - 篇名
  - 摘要
  - 命中类型（JSON/TXT）

### 6.4 ClauseDetail 组件

应继续保留，但职责简化为：

- 展示原文
- 展示白话
- 展示关键词列表

### 6.5 Sidebar 组件

保留当前组件结构，但搜索行为应统一为：

- 搜索条文标题
- 搜索关键词

不再强调图谱节点搜索。

## 7. 检索层改造方案

### 7.1 保留预加载机制

`prefetchKnowledgeBase()` 仍然保留，用于一次性加载配置和可用文件。

### 7.2 新增关系检索函数

建议新增：

```ts
export function resolveClauseRelations(keywords: string[]): RelationHit[]
```

职责：

- 接收条文关键词数组
- 遍历全部关联解析数据源
- 按“优先 JSON，否则 TXT”检索
- 返回统一的 `RelationHit[]`

### 7.3 JSON 优先规则的实现要求

对每个数据源应执行以下逻辑：

1. 如果存在可用 JSON，则只查 JSON
2. 只有当 JSON 文件不存在、加载失败或不可解析时，才查 TXT

不要出现“JSON 命中为空时再继续扫 TXT”的隐式双查，否则会破坏优先级定义。

### 7.4 TXT 输出要求

TXT 命中结果需要包含：

- 标题
- 摘要
- 来源
- 关键词

并限制摘要长度，避免右侧面板和关系图被大段文本淹没。

## 9. 风险与注意事项

### 9.1 关键词质量风险

如果关键词维护不准确，关系图会产生大量误关联。

因此建议：

- 初期只使用人工维护关键词
- 每条条文控制关键词数量
- 优先使用明确术语，避免过泛词

### 9.2 TXT 误命中风险

TXT 全文检索天然会带来噪声。

建议：

- 限制返回数量
- 限制摘要长度
- 使用篇章切分而不是整本匹配

### 9.3 新旧模型并存风险

如果同时保留知识图谱字段和关系图字段，代码容易变得混乱。

建议：

- 短期兼容
- 中期迁移
- 最终删除旧模型依赖

## 11. 结论

本项目的最佳实施路径不是继续补全知识图谱，而是尽快完成向“关键词关系图”的模型切换。

现有仓库已经具备：

- 数据目录基础
- JSON/TXT 双通道检索基础
- 前端三栏布局基础

因此当前最合理的策略是：

1. 保留已有框架
2. 改造条文模型
3. 重写关系生成逻辑
4. 用轻量关系图替代知识图谱

这样改动成本可控，且能更快得到一个真正可用、可维护、可继续扩展的产品版本。
