# 经方关系图需求说明

## 术语对照

为了让不懂英文的用户也能读懂，下文常见英文词按下面理解：

- `JSON（数据文件格式）`
- `TXT（文本文件）`
- `keywords（关键词）`
- `SQLite（轻量数据库）`

## 1. 背景

当前项目以“知识图谱”为核心表达形式，需要对经典条文进行较重的人工语义建模，包括节点类型拆分、节点之间关系定义、图谱层插件挂载等。

这份文档更偏“为什么要改”的说明。

如果你不是开发者，可以把它理解成：

`它主要解释我们为什么不用复杂知识图谱，而改用现在这种关系图。`

该方案存在以下问题：

- 数据建模成本高，依赖人工维护大量 `nodes` 与 `links`
- 新增经典或条文时，录入负担较重
- 图形表达复杂，用户理解门槛较高
- 当前数据规模尚不足以支撑完整知识图谱体系

因此，项目核心展示模型调整为“关系图”，以关键词命中为关系依据，弱化复杂语义建模，提升可维护性与可读性。

## 2. 产品目标

项目不再以“知识图谱”为核心，而改为以“关键词驱动的关系图”为核心展示形式。

系统围绕经典条文中的关键词，自动在关联解析数据源中检索相同关键词，并建立“条文 - 关联文献片段”的关系，用于帮助用户快速浏览经典与相关文献之间的文本联系。

## 3. 核心概念

### 3.1 经典条文

经典条文是关系图的起点节点。每条条文至少包含：

- `id`：条文唯一标识
- `title`：条文标题
- `content`：条文原文
- `translation`：白话解释
- `keywords`：用于建立关系的关键词数组

白话理解：

一条条文最少只要准备好 5 个东西：

1. 编号
2. 标题
3. 原文
4. 白话解释
5. 关键词

### 3.2 关联解析数据源

关联解析数据源是被检索的文本集合，来源于 `data/关联解析/`。

数据源分为两类：

- 结构化 `json`
- 原始文本 `txt`

白话理解：

- `json` 更像已经整理好的“关键词卡片库”
- `txt` 更像还没完全整理的整篇原文

### 3.3 关系

关系的语义定义为：

- 当经典条文中的某个关键词，出现在某个关联解析数据源中时，即建立关联

关系不是病机推理、方药推导或概念本体关系，而是“共享关键词导致的文本关联”。

## 4. 关系图展示模型

关系图建议采用以下最小结构：

- 中心节点：当前经典条文
- 中间节点：条文关键词
- 外围节点：命中的关联文献片段或关联来源

推荐关系链路如下：

`经典条文 -> 关键词 -> 关联文献片段`

不建议直接使用“经典条文 -> 整本典籍”作为唯一关系粒度，否则会导致关系过粗、解释性不足。

## 5. 检索规则

### 5.1 基本规则

对于当前条文的每个关键词，系统按配置遍历关联解析数据源，执行关键词匹配。

只要检索命中，即建立关系。

### 5.2 优先级规则

同一关联数据源的解析顺序固定为：

1. 优先解析 `json`
2. 若不存在可用 `json`，则解析 `txt`

这条非常关键。

你可以直接记成一句话：

`先查整理好的资料，再查原始全文。`

### 5.3 JSON 检索规则

当某个数据源存在对应的 `json` 文件时：

- 优先使用该 `json` 文件
- 按关键词直接读取命中的结构化内容
- 不再继续回退到同源 `txt`，除非该 `json` 文件不存在或不可用

### 5.4 TXT 检索规则

当某个数据源没有可用 `json` 文件时：

- 使用对应 `txt` 文件进行全文检索
- 应按篇章、标题或自然段进行切分
- 命中关键词后返回“篇名/标题 + 命中文本摘要”

不建议在命中后直接返回整本书内容。

## 6. 数据结构要求

### 6.1 经典条文数据

经典条文数据应以“基础信息 + 关键词”为主，不再强制要求维护复杂图谱节点结构。

建议结构如下：

```json
{
  "id": "29",
  "title": "伤寒论 第29条",
  "content": "伤寒脉浮、自汗出……",
  "translation": "太阳病误治后出现变证……",
  "keywords": ["谵语", "吐逆"]
}
```

### 6.2 关联解析配置

关联解析配置需定义数据源名称、文件基名、分类信息等，用于统一调度检索。

建议保留当前配置思路，例如：

```json
[
  {
    "sourceName": "黄帝内经素问",
    "fileBaseName": "437-黄帝内经素问",
    "category": "内经"
  }
]
```

### 6.3 JSON 数据结构

为保证稳定检索，关联解析 `json` 应统一为“按关键词索引”的结构。

如果你看到下面的示例，不要害怕。

它只是表达一个简单意思：

`先写关键词，再把属于这个关键词的相关片段放进去。`

建议结构如下：

```json
{
  "谵语": [
    {
      "title": "热病论",
      "content": "……"
    }
  ],
  "吐逆": [
    {
      "title": "某篇",
      "content": "……"
    }
  ]
}
```

每个命中项至少应包含：

- `title`
- `content`

## 7. 页面行为要求

### 7.1 左侧

- 选择经典
- 选择条文
- 支持按条文标题或关键词搜索

### 7.2 中间主区域

- 展示当前条文原文与白话解释
- 展示当前条文对应的关系图

关系图中应突出：

- 当前条文
- 条文关键词
- 命中的关联来源或片段

### 7.3 右侧

- 展示检索命中的关联内容列表
- 按来源分组
- 展示命中关键词、篇名、摘要内容

## 8. 非功能要求

### 8.1 可维护性

- 数据录入应尽量面向非开发人员
- 新增条文时，应优先通过补充关键词完成关联，而不是手工维护复杂关系边

### 8.2 可扩展性

- 可持续新增新的经典
- 可持续新增新的关联解析来源
- 不应因增加新来源而修改前端核心关系模型

### 8.3 可解释性

- 每条关系都应能说明“因哪个关键词命中”
- 用户应能直观看到关系建立的依据

## 9. 与旧模型的区别

旧模型：

- 强调节点类型建模
- 强调显式 `nodes / links`
- 强调知识图谱语义关系

新模型：

- 强调关键词驱动
- 强调检索命中后的文本关联
- 强调关系可解释、结构轻量、易维护

## 10. 实施建议

第一阶段建议采用“人工维护关键词”的方案，不建议立即引入自动分词或自动抽词。

原因如下：

- 中医文本术语复杂
- 自动提词误差较高
- 错误关键词会直接污染关系结果

因此，短期内推荐流程为：

1. 录入经典条文基础信息
2. 为条文人工维护关键词
3. 配置关联解析来源
4. 按“优先 JSON，否则 TXT”的规则建立关系
5. 前端以关系图而非知识图谱展示结果

## 11. 结论

本项目的核心定位应从“中医经典知识图谱”调整为“中医经典关键词关系图”。

该调整保留了跨文本关联能力，同时显著降低建模复杂度，更符合当前数据基础、维护能力与产品阶段。
