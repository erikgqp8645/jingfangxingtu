# 整理关联解析资料操作手册

## 术语对照

为了让不懂英文的用户也能看懂，下文常见英文词按下面理解：

- `JSON（数据文件格式）`
- `TXT（文本文件）`
- `keywords（关键词）`
- `fileBaseName（文件基础名）`

这份文档只讲一件事：

`怎么给项目添加、整理、维护关联解析资料。`

如果你把经典条文理解成“发出关键词的一方”，那关联解析资料就是“被搜索的一方”。

---

## 1. 先理解关联解析是什么

例如：

经典条文里有关键词：

- 谵语
- 恶寒
- 少阳

系统会去 `data/关联解析/` 里查这些词。

如果某个来源里也有这些词，就会建立关联。

所以，关联解析资料的作用就是：

`给关键词提供出处、解释、原文片段。`

## 4. `guanlianjiexiconfig.json` 怎么写

先记住这一句：

`这个文件不是正文内容，它只是“来源登记表”。`

示例：

```json
[
  {
    "sourceName": "黄帝内经素问",
    "fileBaseName": "437-黄帝内经素问",
    "category": "内经"
  },
  {
    "sourceName": "伤寒寻源",
    "fileBaseName": "467-伤寒寻源",
    "category": "伤寒"
  }
]
```

### 字段解释

#### `sourceName`

页面显示名称。

也就是用户在页面上能看到的名字。

#### `fileBaseName`

真正关联文件名的主名字。

你可以把它理解成“去掉扩展名之后的统一底名”。

例如：

```text
437-黄帝内经素问.json
437-黄帝内经素问.txt
```

它们都对应：

```text
437-黄帝内经素问
```

#### `category`

分类名。

用于分组和显示。

如果你暂时不确定怎么分类，也可以先按书名大类去分。

---

## 5. 文件命名规则

文件名必须和 `fileBaseName` 对应。

例如配置写的是：

```json
{
  "sourceName": "黄帝内经素问",
  "fileBaseName": "437-黄帝内经素问",
  "category": "内经"
}
```

那文件可以是：

- `437-黄帝内经素问.json`
- `437-黄帝内经素问.txt`

不要写成别的名字。

否则系统找不到。

---

## 6. 最推荐的方式：用 JSON

JSON 最适合做精确匹配。

如果你是新手，可以直接把它理解成：

`JSON 更像整理好的词条本，所以命中更准。`

因为它可以直接写成：

`哪个关键词 -> 对应哪段内容`

### JSON 示例

看示例时请只抓住一个重点：

最外层写关键词，里面写对应片段。

```json
{
  "谵语": [
    {
      "title": "病能论篇第四十六",
      "content": "其病发热、汗出、谵语。谵语者，阳气独盛，五脏气之所不营也，故心神狂越而妄言。"
    }
  ],
  "恶寒": [
    {
      "title": "宣明五气篇第二十三",
      "content": "五脏所恶：心恶热，肺恶寒，肝恶风，脾恶湿，肾恶燥，是谓五恶。"
    }
  ]
}
```

---

## 7. JSON 每一层是什么意思

### 最外层键

最外层键就是关键词。

例如：

- `谵语`
- `恶寒`
- `少阳`

### 键后面的数组

表示这个关键词可以有多条对应内容。

### 数组里的对象

每个对象至少要有：

- `title`
- `content`

---

## 8. 什么时候用 TXT

TXT 适合下面两种情况：

1. 你手里是整本原始资料，还没来得及整理成 JSON
2. 你想先把全文放进去，后面慢慢精修

白话说，就是：

`来不及精修时，先把原始全文放进去，也比完全没有资料强。`

### TXT 示例

```text
<目录>卷第七

<篇名>宣明五气篇第二十三

属性：五味所入∶酸入肝，辛入肺，苦入心，咸入肾，甘入脾，是谓五入。
五气所病∶心为噫，肺为咳，肝为语，脾为吞，肾为欠为嚏。
五脏所恶∶心恶热，肺恶寒，肝恶风，脾恶湿，肾恶燥，是谓五恶。
```

系统会把它按段切开，再找关键词。

---

## 9. JSON 和 TXT 同时存在时怎么查

这是现在项目里很重要的一条规则。

新手最容易误会这一条，所以再说一遍：

不是“这个来源只查 JSON（数据文件格式）或者只查 TXT（文本文件）”。

而是：

`同一个来源里，每个关键词先查 JSON，没找到才继续回退查 TXT。`

规则如下：

1. 先查 JSON
2. 如果这个关键词在 JSON 没命中
3. 再去 TXT 查

注意：

这不是“整本来源二选一”。

而是：

`每个关键词单独决定要不要从 TXT 回退。`

## 11. 如果只有 TXT，没有 JSON，怎么办

也可以先用。

例如你有：

[437-黄帝内经素问.txt](C:\Users\hxst01\Documents\aicoding\jingfangxingtu\data\关联解析\437-黄帝内经素问.txt)

只要文件里确实有：

```text
肺恶寒
```

系统也能查到。

但是：

TXT 结果通常不如 JSON 精确。

所以长期看，推荐你把高频关键词慢慢整理进 JSON。

---

## 12. 整理关联解析时最推荐的工作法

### 方法一：先全文，后精修

顺序：

1. 先把原始 TXT 放进去
2. 先让系统能搜索
3. 再把高频关键词一点点整理成 JSON

### 方法二：直接精确整理

顺序：

1. 先决定关键词
2. 直接写 JSON
3. 只保留最有价值的内容

如果你时间有限，推荐方法一。

如果你追求质量，推荐方法二。

---

## 13. 最容易出错的地方

### 错误 1：忘记在配置里登记来源

结果：

文件在目录里，但系统完全不会搜索它。

### 错误 2：文件名和 `fileBaseName` 不一致

结果：

系统找不到正确文件。

### 错误 3：关键词写法不统一

例如：

- 一个地方写 `恶寒`
- 一个地方写 `恶寒者`

这两个不一定会自动当成同一个词。

所以尽量统一关键词写法。

### 错误 4：JSON 结构写错

例如把它写成：

```json
[
  {
    "恶寒": "内容"
  }
]
```

这就不是当前推荐结构。

推荐结构一定是：

```json
{
  "恶寒": [
    {
      "title": "标题",
      "content": "内容"
    }
  ]
}
```

## 14. 什么时候导入数据库

如果你只是刚整理完文件，先刷新页面看效果。

如果你确认这批关系资料稳定了，再执行：

```bash
npm run db:import:relations
```

如果你要全部重建数据库，也可以：

```bash
npm run db:seed
```
