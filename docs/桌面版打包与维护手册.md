# 桌面版打包与维护手册

## 这份手册是做什么的

这份手册专门说明 Windows 桌面版怎么打包、安装后数据放在哪里、以后怎么备份和迁移。

如果你只记住一句话，请记住：

`不要直接改安装目录里的文件，桌面版真正会被修改的数据在用户目录里。`

如果你是第一次打包桌面版，可以先只记住下面 5 步：

1. 先确认当前在桌面版分支
2. 先跑一次代码检查
3. 再生成图标
4. 再打目录版或安装包
5. 打完后先安装和启动检查，不要马上发出去

---

## 1. 适合谁看

这份手册适合下面几类人：

- 你想把项目打包成 Windows 安装包
- 你已经安装了桌面版，想知道数据放在哪里
- 你后面还要继续维护关键词、经典数据和 SQLite 数据库
- 你想把同一份数据迁移到另一台电脑

---

## 2. 桌面版现在怎么工作

桌面版不是简单地把网页包起来，而是包含 3 部分：

- 前端界面
- 本地服务
- 本地可写数据目录

桌面版第一次启动时，会做一件很重要的事：

1. 从安装包里读取内置数据
2. 把这些数据复制到当前 Windows 用户自己的目录
3. 以后真正读写的，都是这个用户目录里的数据

这样做的好处是：

- 安装目录保持干净
- 升级应用时，不容易覆盖用户自己维护的数据
- 后期备份和迁移更简单

---

## 3. 打包命令

建议顺序不是随便选一个命令就跑，而是按下面顺序来。

### 3.0 打包前先检查

先检查这几项：

1. 当前分支是不是 `codex/windows-desktop-app`
2. 页面功能是不是已经能正常运行
3. `npm run lint` 是否通过
4. 是否已经确认本次版本号和安装包命名

如果这 4 项里有没确认的，不建议直接发包。

### 3.1 生成图标资源

```powershell
npm run assets:desktop
```

这个命令会生成：

- `assets/desktop/icon.png`
- `assets/desktop/icon.ico`

源设计文件在：

- `assets/desktop/icon-source.svg`

### 3.2 生成目录版

```powershell
npm run build:desktop:dir
```

输出目录：

- `release/win-unpacked/`

目录版主程序：

- `release/win-unpacked/JingfangRelations.exe`

目录版更适合你自己先做本地检查。

### 3.3 生成安装包

```powershell
npm run build:desktop
```

输出目录：

- `release/`

这里生成的 `.exe` 文件，就是可以分发给别人安装的 Windows 安装包。

如果你准备正式发布，推荐先打目录版自测，再打安装包。

## 4. 安装后数据放在哪里

安装后真正会被修改的数据，不在安装目录里。

默认在 Windows 用户目录下，一般是：

```text
%APPDATA%\经方关系图\workspace\
```

展开后通常类似：

```text
C:\Users\你的用户名\AppData\Roaming\经方关系图\workspace\
```

这个目录里至少会有两类内容：

```text
workspace/
├─ data/
└─ storage/
   └─ app.db
```

含义如下：

- `data/`：可维护的经典 JSON、关联解析 JSON/TXT、配置文件
- `storage/app.db`：SQLite 数据库

也就是说，后面如果用户自己在桌面版里加了关键词、同步了数据库，真正变化的是这里，不是安装目录。

---

## 5. 哪些地方可以改，哪些地方不要改

### 5.1 可以维护的地方

后期维护主要看这两个位置：

- 用户目录里的 `data/`
- 用户目录里的 `storage/app.db`

### 5.2 不建议手工修改的地方

不要把安装目录当成维护入口。

例如不要优先改这些位置：

- `C:\Program Files\...`
- 安装包解压目录里的 `resources`
- `release/win-unpacked/` 里的资源文件

原因很简单：

- 升级时可能被覆盖
- 权限可能受限
- 容易和真实运行数据混淆

---

## 6. 如果要备份，备份什么

最稳妥的做法是直接备份整个用户工作目录：

```text
%APPDATA%\经方关系图\workspace\
```

如果你想最小化备份，至少要备份：

- `data/`
- `storage/app.db`

建议顺序：

1. 先完全退出桌面应用
2. 复制整个 `workspace/` 目录到别的位置
3. 备份文件名里写上日期

例如可以备份成：

```text
workspace-backup-2026-04-23
```

---

## 7. 如果要迁移到另一台电脑

最简单的迁移方法：

1. 在新电脑安装桌面版
2. 先启动一次，再退出
3. 找到新电脑的 `%APPDATA%\经方关系图\workspace\`
4. 用旧电脑备份出来的 `workspace/` 覆盖它
5. 再次启动应用

这样迁移后，一般会同时带过去：

- 经典数据
- 关键词维护结果
- SQLite 数据库内容

---

## 8. 如果要升级桌面版

建议升级顺序：

1. 先备份 `%APPDATA%\经方关系图\workspace\`
2. 再安装新版本
3. 安装完成后检查关键词、条文、关联结果是否正常

因为现在桌面版真正的数据在用户目录里，所以正常升级不应该覆盖你维护过的数据。

但“正常不应该覆盖”不等于“可以不备份”。

正式升级前，最好还是先备份一次。

---

## 9. 常见问题

### 9.1 为什么我改了安装目录里的 JSON，没有效果

因为桌面版实际读写的是用户目录里的 `workspace/data/`，不是安装目录里的内置资源。

### 9.2 为什么我卸载再安装，之前的数据还在

因为数据在 `%APPDATA%` 用户目录，不在安装目录里。

### 9.3 如果我要彻底重置桌面版怎么办

先备份，然后删除：

```text
%APPDATA%\经方关系图\workspace\
```

下次启动时，应用会重新从安装包内置资源复制一份新的基础数据。

---

## 10. 这套桌面结构的优点

现在这套结构对后期维护有几个明显好处：

- 安装包和用户数据分开
- 数据备份路径明确
- 迁移到新电脑简单
- 后面继续加“导入”“修复”“校验”“备份”功能更自然

## 12. 正式发包前的最小检查清单

如果你准备把桌面版发给别人，至少再做一次这 6 条检查：

1. 安装包名称是否正确
2. 安装后桌面图标是否正常
3. 首次启动是否能正常打开
4. 左侧经典目录是否正常显示
5. 选中条文后，正文、关键词、右侧命中是否正常
6. 关闭再打开一次，数据是否仍然保留

只要这 6 条有一条没过，就不建议对外发 Release（发布版本）。
